npm run dev

# Type-check (lint)
npm run lint

npm run build

## Architecture

This is a **credit card sales simulation calculator** — a React SPA backed by a Node/Express server. The server manages authentication against a MySQL database and persists rate configuration to a `config.json` file on disk.

### Multi-Tenant SaaS Pattern

The root project (`src/App.tsx`, `server.ts`) is the **master template**. Each partner gets an isolated copy under `saas/<partner-key>/` (e.g., `saas/d_cred/`, `saas/credfacil/`). The copies are **not hand-maintained** — they are generated by `scripts/copy_to_partners.ps1`, a PowerShell script that:

1. Deletes and recreates each `saas/<partner>` directory from the root source.
2. Does string replacements in `src/App.tsx` to inject the partner's brand name into the browser title, `localStorage` prefix, copyright footer, and the exported PNG header.
3. Injects partner-specific `DEFAULT_TAXAS_CUSTO` rate tables where configured.
4. Copies custom `logo.png` and `favicon.png` from `saas_assets/<partner>/` into each tenant's `public/` directory.

**Rule**: Any logic change to `src/App.tsx` or `server.ts` must be followed by running `copy_to_partners.ps1` and rebuilding all tenants before deploying.

### LocalStorage Isolation

Each tenant has its own `STORAGE_PREFIX` constant (e.g., `"emprestabh_"` for the base, `"d_cred_"` for D Cred). All `localStorage` reads and writes go through `getStorageItem`/`setStorageItem` helpers that prepend this prefix. This prevents settings from bleeding across tenants when they share the same browser.

### Rate Configuration Flow

Rates (`fatores`, `acrescimos`, `taxas_custo`) live in two places simultaneously:
- **`localStorage`**: Loaded on mount as a fast client-side cache.
- **`api/config/` endpoint**: On login, the frontend GETs the server's `config.json` and overwrites the local cache. When the `dono` role saves from the Admin panel, a POST syncs back to the server.

In production on cPanel, `api/config/index.php` handles reads/writes of `config.json` on disk. In local development, `server.ts` mirrors this behavior using Node's `fs` to read/write `config.json` at the project root.

### User Roles

- **`dono`**: Full access. Sees the Admin panel (gear icon), can edit rates, visibility flags, logo, and primary color. Auto-authenticated into admin settings on login.
- **`vendedor`**: Simulation only. No admin gear visible. The "Lucro Líquido" column visibility is controlled remotely by the `dono` via `show_lucro_vendedor`.

Authentication is handled by `POST /api/login/`. Credentials are validated against MySQL first, with hardcoded fallback credentials (`dono/123456`, `vendedor/123456`) for offline/dev mode.

### Calculation Engine

The simulation table (`simulacao`) is a `useMemo` in `App.tsx` that computes 21 installment rows. For each row `i`:

```
fator = fatoresBase[i] + acrescimoNivel + acrescimoGeral
totalAPassar = valorDesejado * fator  (when mode = "valor")
valorParcela = totalAPassar / i
lucro = totalAPassar - custoMaquina - valorLiquido
```

The two table types ("Normal" and "Promo") use separate `fatoresBase` and `acrescimos` dictionaries.

### Deployment

GitHub Actions (`.github/workflows/deploy.yml`) triggers on push to `main`. It builds the root project and all 10 tenant projects sequentially, deploying each via FTP to separate directories on the same cPanel shared host:
- Root → `./` (calculadora base)
- `saas/d_cred/dist/` → `./d_cred/`
- `saas/credpara/dist/` → `./credpara/`
- etc.

FTP credentials are stored as GitHub secrets: `FTP_SERVER`, `FTP_USERNAME`, `FTP_PASSWORD`.

### Environment Variables

Copy `.env.example` to `.env` and fill in:
- `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASSWORD` — MySQL credentials for the cPanel database.
- `PORT` — defaults to 3000.
- `NODE_ENV` — set to `production` on the server.

### Adding a New Partner

1. Add partner assets (logo + favicon) to `saas_assets/<partner-key>/`.
2. Add the partner entry to the `$partners` hash and optionally `$partnerTaxasCusto` in `scripts/copy_to_partners.ps1`.
3. Run the PowerShell script to generate `saas/<partner-key>/`.
4. Add build and FTP deploy steps to `.github/workflows/deploy.yml`.

### Common Pitfalls (from `ai_context/erros_comuns.md`)

- **Git push blocked by large files**: `node_modules` or `.zip` files leaked into git tracking. Fix: clean `.gitignore`, run `git rm -r --cached .`, and reindex.
- **ZIP generation is slow on Windows**: Use `tar -a -c -f archive.zip *` instead of `Compress-Archive`. To exclude folders: `tar --exclude="node_modules" --exclude=".git" -a -c -f name.zip *`.
- **Partner brand name in dynamic exports**: Extract partner name from `document.title.split(' - ')[0]` — don't hardcode strings that need per-tenant replacement.
